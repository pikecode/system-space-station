## 一、技术栈

| 层 | 选型 | 说明 |
|---|---|---|
| 前端 | React 18 + TypeScript | 类型安全，组件生态成熟 |
| 前端 UI | Ant Design 5.x + @ant-design/pro-components | antd 组件库 + ProTable/ProForm 等高级组件，权限路由自行实现 |
| 前端路由 | React Router v6 | 嵌套路由支持角色布局隔离，结合 loader 做权限拦截 |
| 前端状态 | Zustand | 轻量，适合中小型后台 |
| 前端请求 | TanStack Query + Axios | 缓存、loading、错误处理统一管理 |
| 后端框架 | NestJS + TypeScript | 模块化，依赖注入，适合 RBAC 权限体系 |
| 数据库 | PostgreSQL | 事务支持强，分成计算原子性有保障 |
| ORM | Prisma | 类型安全，迁移管理清晰 |
| 认证 | JWT（Access Token） | 无状态，前后端分离友好 |
| 小程序端 | Taro 4.x + React + TypeScript | 与 Web 同语言，共享 packages/shared 类型，支持微信小程序 / H5 双端 |
| 工程管理 | pnpm monorepo | 前端、小程序、后端共享 TypeScript 类型定义 |

## 四、数据模型

### 4.1 用户与组织

```prisma
model Department {
  id          String     @id @default(cuid())
  name        String                        // 部门名称
  description String?                       // 部门描述（可选）
  headId      String?    @unique            // 部门负责人 userId，@unique 保证1:1，null 表示暂无负责人
  sort        Int        @default(0)        // 排序权重，数字越小越靠前
  status      DeptStatus @default(ACTIVE)   // ACTIVE=正常 INACTIVE=已停用
  createdAt   DateTime   @default(now())    // 创建时间
  updatedAt   DateTime   @updatedAt         // 最后更新时间

  head        User?      @relation("DeptHead", fields: [headId], references: [id])
  members     User[]     @relation("DeptMembers")
  customers   Customer[]
}

model Position {
  id        String         @id @default(cuid())
  name      String                              // 岗位名称（如"销售顾问""区域经理"）
  sort      Int            @default(0)          // 排序权重
  status    PositionStatus @default(ACTIVE)     // ACTIVE=启用 INACTIVE=停用
  createdAt DateTime       @default(now())
  updatedAt DateTime       @updatedAt

  users     User[]
}

model User {
  id           String     @id @default(cuid())
  name         String                        // 姓名
  phone        String     @unique            // 手机号，同时作为登录账号
  email        String?                       // 邮箱（可选）
  avatar       String?                       // 头像 URL
  positionId   String?                       // 岗位 ID，关联 Position 表（下拉选择）
  passwordHash String                        // 密码哈希，bcrypt 加密
  role         UserRole                      // ADMIN=系统管理员 HEAD=部门负责人 MEMBER=部门成员
  departmentId String?                       // 所属部门 ID，ADMIN 角色可为 null
  authVersion  Int        @default(1)        // Token 版本号；调岗/禁用时 +1，使旧 JWT 失效
  status       UserStatus @default(ACTIVE)   // ACTIVE=在职 INACTIVE=已离职/禁用
  lastLoginAt  DateTime?                     // 最后登录时间
  createdAt    DateTime   @default(now())    // 账号创建时间
  updatedAt    DateTime   @updatedAt         // 最后更新时间

  position                  Position?          @relation(fields: [positionId], references: [id])
  department                Department?        @relation("DeptMembers", fields: [departmentId], references: [id])
  headOf                    Department?        @relation("DeptHead")
  customers                 Customer[]         @relation("AssignedCustomers")
  createdCustomers          Customer[]         @relation("CreatedCustomers")
  submittedMemberships      Membership[]       @relation("MembershipSubmitter")
  reviewedMemberships       Membership[]       @relation("MembershipReviewer")
  refundReviewedMemberships Membership[]       @relation("MembershipRefundReviewer")
  receivedCommissions       CommissionRecord[] @relation("CommissionReceiver")
}

enum PositionStatus { ACTIVE INACTIVE }      // 启用 | 停用

enum UserRole   { ADMIN HEAD MEMBER }       // 系统管理员 | 部门负责人 | 部门成员
enum UserStatus { ACTIVE INACTIVE }         // 在职 | 离职/禁用
enum DeptStatus { ACTIVE INACTIVE }         // 正常 | 已停用
```

### 4.2 客户

> 客户分两类：**个人客户**和**企业客户**，用 `customerType` 区分，字段按类型选填。

```prisma
model Customer {
  id           String         @id @default(cuid())
  customerType CustomerType   @default(INDIVIDUAL)    // 客户类型：INDIVIDUAL=个人 COMPANY=企业

  // ── 通用字段（个人/企业均有）──────────────────────────────
  name         String                                  // 显示名称：个人填姓名，企业填公司名称
  phone        String                                  // 主联系电话
  source       CustomerSource @default(OTHER)          // 来源渠道
  tags         String?                                 // 标签，逗号分隔（如"高净值,转介绍"）
  notes        String?                                 // 备注信息
  wechat       String?                                 // 微信号

  // ── 个人客户专用字段 ──────────────────────────────────────
  gender       Gender?                                 // 性别（个人客户）
  birthday     DateTime?                               // 生日（个人客户，用于节日关怀）
  address      String?                                 // 联系地址（个人客户）
  // 注：身份证字段一期不收集（YAGNI），如后期需要须加密存储并脱敏展示

  // ── 企业客户专用字段 ──────────────────────────────────────
  creditCode   String?                                 // 统一社会信用代码（企业客户）
  industry     String?                                 // 所属行业（企业客户）
  contactName  String?                                 // 企业联系人姓名
  contactPhone String?                                 // 企业联系人手机（与公司电话可不同）

  // ── 归属字段 ───────────────────────────────────────────────
  assignedTo   String                                  // 当前维护人 userId（1客户:1维护人）
  createdBy    String                                  // 录入人 userId（不随维护人变更而变）
  departmentId String                                  // 所属部门 ID（随维护人变更同步更新）
  status       CustomerStatus @default(ACTIVE)         // ACTIVE=正常 INACTIVE=已停用
  createdAt    DateTime       @default(now())          // 录入时间
  updatedAt    DateTime       @updatedAt               // 最后更新时间

  assignedUser  User        @relation("AssignedCustomers", fields: [assignedTo], references: [id])
  creator       User        @relation("CreatedCustomers", fields: [createdBy], references: [id])
  department    Department  @relation(fields: [departmentId], references: [id])
  memberships   Membership[]

  @@index([assignedTo])
  @@index([departmentId])
  @@index([status])
}

enum CustomerType   { INDIVIDUAL COMPANY }             // 个人客户 | 企业客户
enum Gender         { MALE FEMALE UNKNOWN }            // 男 | 女 | 未知
enum CustomerSource {
  REFERRAL        // 转介绍
  SELF_DEVELOPED  // 自主开发
  ACTIVITY        // 活动获客
  ONLINE          // 线上渠道
  OTHER           // 其他
}
enum CustomerStatus { ACTIVE INACTIVE }                // 正常 | 已停用
```

```prisma

  endDate          DateTime                               // 会员有效期结束日

  reviewNote       String?                                // 审批备注（拒绝时必填原因）
  refundReason     String?                                // 退款原因（申请退款时必填）
  refundReviewedBy String?                                // 退款审批人 userId（由 HEAD 审批）
  refundReviewer   User?            @relation("MembershipRefundReviewer", fields: [refundReviewedBy], references: [id])
  refundReviewedAt DateTime?                              // 退款审批时间
  refundAt         DateTime?                              // 退款完成时间
  createdAt        DateTime         @default(now())       // 申请提交时间
  updatedAt        DateTime         @updatedAt            // 最后更新时间

  customer      Customer           @relation(fields: [customerId], references: [id])

  commissions   CommissionRecord[]

  @@index([customerId])
  @@index([status, endDate])
  @@index([submittedBy])
}

enum MemberLevelStatus { ACTIVE INACTIVE }             // 启用 | 停用
enum MembershipStatus {
  PENDING        // 待审核：已提交，等待负责人审批
  APPROVED       // 有效：审批通过，分成已生成
  REJECTED       // 已拒绝：审批不通过，可修改后重新提交
  EXPIRED        // 已到期：有效期结束
  REFUND_PENDING // 退款申请中：客户申请退款，等待确认
  REFUNDED       // 已退款：退款完成，已生成负向冲账记录
```

### 4.4 分成

```prisma
model CommissionConfig {
  id             String   @id @default(cuid())
  memberRatio    Decimal  @db.Decimal(5, 2)     // 维护人员分成比例（%），三项之和必须 = 100
  headRatio      Decimal  @db.Decimal(5, 2)     // 部门负责人分成比例（%）
  companyRatio   Decimal  @db.Decimal(5, 2)     // 公司分成比例（%）
  settlementDays Int      @default(15)           // 结算周期天数（1-365，默认 15 天）
  effectiveFrom  DateTime                        // 生效时间；取最新生效配置：effectiveFrom <= NOW() ORDER BY effectiveFrom DESC, createdAt DESC LIMIT 1
  remark         String?                         // 配置备注（如"2026年Q3调整"）
  createdBy      String                          // 操作人 userId
  createdAt      DateTime @default(now())        // 配置创建时间
  // 注：此表为不可变版本表，每次修改 INSERT 新行，绝不 UPDATE 旧行；历史分成快照依赖旧行

  @@index([effectiveFrom])                       // close-period 和审批时查最新配置
}

model CommissionRecord {
  id               String           @id @default(cuid())
  businessKey      String           @unique                 // 业务唯一键：原分成="earn:{membershipId}:{role}"，冲账="refund:{membershipId}:{role}"；防止重复生成
  membershipId     String                                   // 关联的会员记录 ID
  entryType        CommissionEntry  @default(EARNING)       // EARNING=正向分成 REVERSAL=冲账（退款时生成）
  receiverUserId   String?                                  // 收款方 userId；COMPANY 角色时为 null
  receiverUser     User?            @relation("CommissionReceiver", fields: [receiverUserId], references: [id])
  receiverRole     CommissionRole                           // 分成角色：MEMBER=维护人 HEAD=负责人 COMPANY=公司
  amount           Decimal          @db.Decimal(10, 2)     // 分成金额（元）；冲账记录为负数
  ratio            Decimal          @db.Decimal(5, 2)      // 分成比例快照（审批时固化，不随配置修改变化）
  status           CommissionStatus @default(PENDING)      // 见枚举；冲账记录同样经历待结算→待出账→已结算
  periodId         String?                                  // 归属结算周期 ID，未归入周期时为 null
  originalRecordId String?                                  // 冲账记录指向的原分成记录 ID（正向记录为 null）
  remark           String?                                  // 备注（如冲账原因）
  createdAt        DateTime         @default(now())         // 分成记录生成时间
  settledAt        DateTime?                                // 财务结算确认时间

  membership     Membership         @relation(fields: [membershipId], references: [id])
  period         SettlementPeriod?  @relation(fields: [periodId], references: [id])
  originalRecord CommissionRecord?  @relation("Reversal", fields: [originalRecordId], references: [id])
  reversals      CommissionRecord[] @relation("Reversal")

  @@index([membershipId])
  @@index([status, periodId])
}

model SettlementPeriod {
  id             String       @id @default(cuid())
  startDate      DateTime                          // 周期开始日期
  endDate        DateTime                          // 周期结束日期
  totalFee       Decimal      @db.Decimal(12, 2)  // 本周期内会员费总额（close-period 时固化）
  totalAmount    Decimal      @db.Decimal(12, 2)  // 本周期分成总额（close-period 时固化）
  recordCount    Int          @default(0)          // 归集的分成明细条数（用于结算时校验数量一致性）
  status         PeriodStatus @default(OPEN)       // OPEN=进行中 PENDING_PAYMENT=待出账 SETTLED=已结算
  settledBy      String?                           // 财务确认操作人 userId
  settledAt      DateTime?                         // 财务确认结算时间
  remark         String?                           // 备注（如"7月第一期"）
  createdAt      DateTime     @default(now())      // 周期创建时间

  records        CommissionRecord[]

  @@index([status, endDate])                       // 定时任务扫描 OPEN 周期用
}

enum CommissionEntry  { EARNING REVERSAL }             // 正向分成 | 冲账（退款）
enum CommissionRole   {
  MEMBER    // 维护人员
  HEAD      // 部门负责人
  COMPANY   // 公司
}
enum CommissionStatus {
  PENDING          // 待结算：会员有效，等待周期汇总（EARNING/REVERSAL 均适用）
  PENDING_PAYMENT  // 待出账：已归入结算周期，等待财务确认
  SETTLED          // 已结算：财务已确认
}
// 注：冲账语义由 entryType=REVERSAL + amount<0 表达，不再用独立 REVERSED 状态
enum PeriodStatus {
  OPEN             // 进行中：当前结算周期
  PENDING_PAYMENT  // 待出账：周期结束，等待财务确认
  SETTLED          // 已结算：财务已完成对账
}
```

## 五、事务与幂等设计

### 审批通过（核心事务）

```
前置：读取当前生效配置 → effectiveFrom <= NOW() ORDER BY effectiveFrom DESC, createdAt DESC LIMIT 1

BEGIN TRANSACTION
  1. WHERE id = :id AND status = 'PENDING'  → UPDATE Membership.status = 'APPROVED'
     （条件更新防并发重复审批；受影响行数 = 0 则回滚返回 409）
  2. 计算分成金额（见金额精度规则）；若维护人 = 当前部门负责人，HEAD 金额 = 0，差额归公司
  3. INSERT CommissionRecord × 3，businessKey = "earn:{membershipId}:{role}"
     （businessKey @unique 约束防止重复插入，冲突则回滚返回 409）
COMMIT
```

### 退款冲账（事务）

冲账记录必须**逐条复制**原分成快照，不能根据当前维护人/配置重新计算：

```
BEGIN TRANSACTION
  1. WHERE id = :id AND status = 'REFUND_PENDING'  → UPDATE Membership.status = 'REFUNDED'
  2. 查询原3条 CommissionRecord WHERE membershipId = :id AND entryType = 'EARNING'
  3. 对每条原记录 INSERT 冲账记录：
       receiverUserId   = original.receiverUserId
       receiverRole     = original.receiverRole
       ratio            = original.ratio
       amount           = -original.amount          // 取负
       entryType        = REVERSAL
       originalRecordId = original.id
       businessKey      = "refund:{membershipId}:{role}"
COMMIT
```

### 周期关闭（close-period 定时事务）

系统同一时刻只允许一个 OPEN 周期；新建部署时系统初始化创建第一个周期。

```
BEGIN TRANSACTION（advisory lock 保证单实例执行）
  1. SELECT ... FOR UPDATE WHERE status = 'OPEN' AND endDate <= NOW()
     （无匹配行则 SKIP，不报错）
  2. 查询 CommissionRecord WHERE status = 'PENDING' AND periodId IS NULL
     → 归集到本周期：UPDATE periodId = :periodId，status = 'PENDING_PAYMENT'
  3. 汇总 totalFee、totalAmount、recordCount 写入 SettlementPeriod
  4. UPDATE SettlementPeriod.status = 'PENDING_PAYMENT'
  5. INSERT 新 SettlementPeriod（OPEN，startDate = 今天，endDate = 今天 + settlementDays）
     （settlementDays 从 effectiveFrom <= NOW() 最新配置取）
COMMIT
```

### 结算确认（财务手动触发事务）

```
BEGIN TRANSACTION
  1. WHERE id = :periodId AND status = 'PENDING_PAYMENT'
     → UPDATE SettlementPeriod.status = 'SETTLED'，settledBy / settledAt
     （受影响行数 = 0 则回滚，防重复结算）
  2. WHERE periodId = :periodId AND status = 'PENDING_PAYMENT'
     → UPDATE CommissionRecord.status = 'SETTLED'，settledAt = NOW()
  3. 校验：步骤2更新行数 = SettlementPeriod.recordCount，不一致则回滚
COMMIT
```

### 金额精度与舍入规则

```
维护人分成 = FLOOR(fee × memberRatio / 100, 2)   // 向下取整到分
负责人分成 = FLOOR(fee × headRatio  / 100, 2)    // 向下取整到分
公司分成   = fee - 维护人分成 - 负责人分成        // 差额项，吸收舍入误差
```

公司分成作为差额项，确保三项之和严格等于会员费，避免每笔单独舍入导致合计差额。

**负责人自维护规则**：当维护人 `assignedTo` = 当前部门负责人 `Department.headId` 时，HEAD 层分成金额置 0，差额自动归入公司分成（公司 = fee − 维护人分成 − 0）。三条记录仍然生成，HEAD 记录 amount = 0、ratio 保留快照值。

**防重复执行**：任务执行前用 PostgreSQL advisory lock（`pg_try_advisory_lock`）+ 条件更新，确保单次执行；单实例一期首选此方案，无需引入额外中间件。多实例部署时改用 `pg-boss`。

**失败补偿**：任务执行结果写入 `SchedulerLog` 表（见数据模型 4.5），记录任务名、执行时间、状态、影响行数，失败时告警并支持手动重跑。

**时区**：所有定时任务以服务器配置时区为准，建议统一设置 `TZ=Asia/Shanghai`。

## 七、权限设计（RBAC）

### 角色权限矩阵

```
ADMIN  → 所有接口
HEAD   → 本部门客户读写 + 审批接口 + 本部门报表
MEMBER → 本人名下客户读写 + 提交会员申请 + 个人分成查看
```

### 实现方式

NestJS `@Roles()` 装饰器 + `RolesGuard`，JWT payload 携带 `role`、`departmentId`、`authVersion`；每次请求从数据库读取当前用户状态，与 Token 中的 `authVersion` 对比。

### Token 失效策略

用户调岗、角色变更或被禁用时，将 `User.authVersion` 自增 1；JWT 中携带签发时的 `authVersion`，`RolesGuard` 将其与数据库当前值对比，不一致则返回 401，强制重新登录。

### 调岗事务规则

用户调岗（`departmentId` 变更）时：

```
前提校验：
  - MEMBER 调岗：无额外限制
  - HEAD 调岗：目标部门不能已有负责人（headId @unique 在 DB 层保证）

调岗事务（同一 DB 事务）：
  1. 若为 HEAD：旧部门 headId → null（清除负责人）
  2. BATCH UPDATE Customer.departmentId = 新部门 WHERE assignedTo = :userId
  3. UPDATE User.departmentId = 新部门
  4. 若调岗角色仍为 HEAD：新部门 headId = :userId
  5. User.authVersion += 1（使旧 Token 立即失效）

禁用 / HEAD 降级为 MEMBER 时同样需要：
  - 清除该用户所在部门的 headId
  - （禁用时）可选：客户自动分配给负责人或挂起待重新分配
```

> 管理员操作界面须展示"该用户名下 N 位客户将一同迁移至新部门"确认提示。

```
JWT payload: { userId, role, departmentId, authVersion, exp }

RolesGuard 校验流程：
1. 验证 JWT 签名和过期时间
2. 从数据库读取 User.authVersion 和 User.status
3. payload.authVersion !== db.authVersion → 401
4. db.status === INACTIVE → 401
5. 通过 → 继续角色和数据行级校验
```

> Access Token 建议有效期设为 2 小时，缩短旧 Token 的暴露窗口。
