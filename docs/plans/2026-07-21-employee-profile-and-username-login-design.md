# 员工档案与管理员用户名登录设计

## 目标

在现有 `User` 模型上补充员工档案字段，并允许系统管理员使用用户名或手机号登录，同时保持现有员工手机号登录兼容。

## 方案

- 用户名为可选唯一字段，只允许管理员通过用户管理服务维护；新建管理员必须填写用户名。
- 员工档案增加工号、性别、出生日期、备用电话、微信和地址。工号唯一，新建员工必须填写。
- 身份证号不保存明文。API 使用独立的 `PII_ENCRYPTION_KEY` 执行 AES-256-GCM 加密，并保存 HMAC-SHA256 指纹用于重复校验，同时保存掩码用于管理页面展示。
- 用户列表、详情和审计日志不返回或记录身份证密文、指纹与完整号码。编辑时身份证输入留空代表不修改。
- 登录接口优先读取 `account`，同时兼容旧客户端的 `phone` 字段；认证失败统一返回“账号或密码错误”，避免泄露账号是否存在。

## 取舍

- 不提供身份证搜索、解密回显和密钥轮换界面，避免扩大一期范围（YAGNI）。
- 不拆分独立员工档案表，当前字段与用户生命周期一致，继续使用模块化单体更简单（KISS）。
- 加密、指纹和掩码集中在 `SensitiveDataService`，避免业务服务重复密码学逻辑（DRY/SOLID）。

## 验证

- 单元测试覆盖密文随机性、可解密性、稳定指纹、用户脱敏选择和用户名登录。
- 应用迁移后执行 Prisma 校验、测试、Lint、类型检查、生产构建。
- 浏览器验证管理员用户名登录、管理员维护用户名和员工档案新增/编辑表单。
